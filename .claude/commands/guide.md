# Guide - 사용법 빠른 안내

> 빠른 모델(haiku)로 즉시 응답

## 역할

에이전트와 스킬 사용법을 빠르게 안내합니다.

## 📋 스킬 목록 (슬래시 명령어)

### 워크플로우 스킬
| 스킬 | 용도 | 예시 |
|------|------|------|
| `/feature` | 새 기능 개발 | `/feature 필터 기능 추가` |
| `/bugfix` | 버그 수정 | `/bugfix 차트 안 그려져` |
| `/refactor` | 리팩토링 | `/refactor PassSchedulePage.vue` |
| `/optimize` | 성능 최적화 | `/optimize icdStore` |
| `/cleanup` | 코드 정리 | `/cleanup console` |
| `/done` | 작업 완료 + 커밋 | `/done` |

### 관리/분석 스킬

| `/health` | 빌드/품질 점검 | `/health` |

## 🚀 일상 패턴

```
작업 시작
└── /health (빌드/상태 확인)

새 기능 개발
├── /plan (계획 수립)
├── /feature (워크플로우 시작)
├── 구현 (fe-expert / be-expert)
└── /done (품질점검 + 완료 + 커밋)

버그 수정
├── /bugfix (워크플로우 시작)
├── 분석 + 수정
└── /done

리팩토링
├── /refactor (파일 분리)
├── /optimize (성능 개선)
├── /cleanup (코드 정리)
└── /done

상태 확인
├── /status (현황)
├── /health (빌드/품질)
└── /sync (문서 동기화 - 별도 실행 가능)
```

**참고:** `/done`은 자동으로 `/sync`를 먼저 실행합니다. `/sync`만 별도 실행도 가능.

## 💡 자주 쓰는 명령

```
# 프로젝트 상태 한눈에
/health

# console.log 정리
/cleanup console

# icdStore 최적화
/optimize icdStore

# 대형 파일 분리
/refactor PassSchedulePage.vue

# 작업 완료 후 커밋
/done
```

## 🤝 협업 대화 패턴

> AI와 효과적으로 협업하는 방법

### 기본 원칙

1. **모든 작업 전 확인** - AI가 먼저 이해한 내용을 확인
2. **양방향 의견 교환** - AI도 의견 제시, 사용자도 피드백
3. **단계별 진행** - Research → Plan → Implement

### 작업 시작 시 AI가 확인하는 것

```
사용자: "icdStore 최적화 해줘"

AI 응답 패턴:
┌─────────────────────────────────────────────────┐
│ 📋 요청 분석                                      │
│                                                 │
│ **이해한 내용:**                                  │
│ - icdStore.ts 성능 개선                          │
│ - 현재 4,500줄 → 분리 또는 최적화                 │
│                                                 │
│ **확인 필요:**                                    │
│ 1. 주요 목표가 뭔가요?                            │
│    □ 파일 크기 줄이기 (분리)                      │
│    □ 렌더링 성능 개선                            │
│    □ 메모리 사용량 줄이기                         │
│                                                 │
│ 2. 우선순위가 있나요?                             │
│    (예: "일단 분리만" 또는 "전부 다")             │
└─────────────────────────────────────────────────┘
```

### AI 의견 제시 패턴

```
사용자: "상태 관리를 어떻게 하면 좋을까?"

AI 응답 패턴:
┌─────────────────────────────────────────────────┐
│ 💡 내 의견                                       │
│                                                 │
│ **추천: Pinia Setup Store + Composable 분리**    │
│                                                 │
│ 이유:                                           │
│ 1. ACS 기존 패턴과 일치                          │
│ 2. TypeScript 타입 추론 우수                     │
│ 3. 테스트 용이                                   │
│                                                 │
│ **대안들:**                                      │
│ | 방식 | 장점 | 단점 |                           │
│ | Options API | 익숙함 | 타입 약함 |             │
│ | Vuex | 레거시 지원 | 복잡함 |                  │
│                                                 │
│ 어떻게 생각하세요?                               │
└─────────────────────────────────────────────────┘
```

### 효과적인 요청 예시

#### ✅ 좋은 요청
```
"Pass Schedule 페이지에 차트 줌 기능 추가해줘.
 - 마우스 휠로 확대/축소
 - 드래그로 이동
 - 기존 Quasar 차트 활용"
```
→ 구체적, 범위 명확, 기술 힌트 제공

#### ⚠️ 개선 가능한 요청
```
"차트 좀 고쳐줘"
```
→ AI가 확인 질문을 함:
- 어떤 차트? (Pass Schedule? Ephemeris?)
- 무엇을 고쳐야? (버그? 기능 추가?)

### 단계별 작업 흐름

```
[1단계: 요청]
사용자 → 작업 요청

[2단계: 확인]  ← AI가 먼저 확인
AI → "이렇게 이해했는데 맞나요?"
   → "추가로 알아야 할 것: ..."

[3단계: 계획]
AI → 작업 계획 제시
   → "이 방향이 좋을 것 같은데, 어떻게 생각하세요?"

[4단계: 합의]
사용자 → 승인/수정 요청

[5단계: 실행]
AI → 구현 진행
   → 중간 체크포인트에서 확인

[6단계: 완료]
AI → 결과 보고 + 다음 단계 안내
```

### 모르면 물어보기

AI가 정보가 부족할 때:
```
"이 부분은 제가 확신이 없어서 확인이 필요해요:
 - XXX 파일의 YYY 로직이 현재 어떤 상태인지
 - 기존에 시도했던 방법이 있는지

 알려주시면 더 정확하게 도움드릴 수 있어요."
```

### 의견 충돌 시

```
사용자: "A 방식으로 하고 싶어"
AI: "A도 가능하지만, B가 더 나을 수 있어요.

     A 방식:
     + 장점1
     - 단점1 (이게 좀 걸려요)

     B 방식:
     + 장점1
     + 장점2

     최종 결정은 사용자님께서 해주세요.
     A로 진행할까요?"
```

## 🎯 상황별 추천 명령어

| 상황 | 추천 | 이유 |
|------|------|------|
| 뭘 해야 할지 모르겠음 | `/status` | 현재 진행 상황 파악 |
| 새 기능 시작 | `/plan` → `/feature` | 계획 먼저, 구현 나중 |
| 버그 발견 | `/bugfix` | 원인 분석부터 체계적으로 |
| 코드가 지저분함 | `/cleanup` | console.log 등 정리 |
| 작업 끝남 | `/done` | 문서 동기화 + 커밋 |
| 빌드 안 됨 | `/health` | 전체 상태 점검 |
